# 技术方案核验规则

## 目录

1. 原子要求
2. 主结论
3. 问题标签
4. 风险等级
5. 核验维度
6. 图文核验
7. 修改建议

## 1. 原子要求

把能够独立满足、缺失或违反的约束拆成独立要求。优先按以下边界拆分：

- 编号条款、列表项和表格行；
- “并且、同时、分别、至少、不得、应支持”连接的独立约束；
- 独立的数量、性能、兼容性、接口、安全、实施或验收指标。

不要为了数量机械拆句。共同描述一个不可分割能力的条件保留为一条要求。

要求类型使用：功能、性能、架构、接口、部署、安全、兼容性、实施、培训、运维、验收、交付物、其他技术要求。

## 2. 主结论

每条原子要求只能有一个主结论：

- **充分响应**：明确覆盖全部约束，内容与采购要求一致，并给出足以理解或验证的实现、边界或落实路径。
- **部分响应**：存在相关内容，但遗漏细项、指标、边界、实施方式、验证方式或关键前提。
- **未响应**：未找到有效响应，或只有与要求无关的泛化描述。
- **无法判断**：来源缺失、表述含糊、图片不可读、表格损坏或依赖未提供附件，导致无法可靠判断。

“支持、满足、兼容”之类的单句表态通常不能单独证明充分响应。原样复述采购要求也不能自动视为充分响应。

## 3. 问题标签

可同时使用多个标签：

- 冲突
- 超范围风险
- 顺序偏离
- 图文不一致
- 证据不足
- 疑似复制残留
- 需人工确认

标签不能替代主结论。

## 4. 风险等级

- **高风险**：可能导致否决、重大扣分、明确负偏离、关键指标不满足、重大承诺冲突或方案不可实施。
- **中风险**：可能造成响应不充分、评审定位困难、实施边界不清或后续争议，通常可通过补写或调整解决。
- **低风险**：主要影响可读性、定位性、一致性或专业表达，对实质响应影响较小。

凡不是“充分响应且无问题标签”，都必须给出风险等级和理由。风险等级依据影响，不依据措辞激烈程度。

## 5. 核验维度

### 完整性

逐条寻找响应，不得因一章提到相关主题就推定其所有细项均覆盖。一条要求对应多处响应时列出所有实质性支撑该结论的出处；不得因页码或文字重复而只保留一处。

### 充分性

区分四层：复述要求、仅声明支持、给出部分方案、给出完整且可验证的落实路径。说明缺失的是哪一层内容。

### 顺序与可定位性

章节顺序不同不等于未响应。以下情况标记顺序偏离：响应过度分散、标题差异过大且无索引、关键内容位于不符合阅读预期的位置、评委需跨多章拼接才能判断。

### 来源定位

采购与标书两侧都显示 Word 物理页码、章节路径、来源 ID 和最多 100 字的连续原文。跨页显示完整页码范围；映射有歧义时显示全部候选页码；无可靠映射时明确写“页码无法确认”。页码不能替代来源 ID，来源 ID 也不能替代页码。

### 冲突

检查文字-文字、文字-表格、文字-图片、同模块、跨模块、标书-采购要求六类冲突。数值单位、数量、产品名、部署方式、接口、周期、SLA、人员和交付物优先核对。

### 超范围内容

采购未要求的增强项不自动构成风险。仅当它引入额外交付或费用、扩大验收责任、增加外部依赖、造成安全/合规/兼容风险，或与其他承诺冲突时标记风险。

## 6. 图文核验

检查架构图、部署图、流程图、产品截图、图片化表格和参数图。自动排除页眉页脚图；对正文中的装饰候选仍需保守确认。

逐图比较：

- 产品、客户、项目和品牌名称；
- 模块层级、组件关系、数据流；
- 数量、性能、容量、版本和参数；
- 流程步骤、角色、责任和异常分支；
- 图片是否支撑邻近文字；
- 图片是否形成采购文件未要求的额外承诺。

不得仅凭文件名、缩略图尺寸或邻近题注判断图片内容。可读图片必须实际打开查看。

## 7. 修改建议

建议类型包括补写、改写、删除、调整位置、增加交叉引用、人工确认。建议必须指出目标位置和需要补充的要素。

有事实依据时可提供参考文本；缺少依据时只给“待确认的补写框架”，不得虚构指标、工具品牌、人员数量、周期、接口或服务承诺。
